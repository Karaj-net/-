## Основной текст
### 1 Слайд
Паттерн проектирования **Строитель (Builder)** — это порождающий паттерн, который позволяет создавать сложные объекты пошагово. Он отделяет процесс конструирования объекта от его конечного представления, что дает возможность использовать один и тот же процесс сборки для получения различных вариантов продукта.
### Слайд 2: Проблема: Создание сложных объектов
#### 1.Пример: объект `Дом` (фундамент, стены, крыша, окна, двери...)
Представим, что в нашей программе мы моделируем объект `Дом`. Дом — это не простое целое число или строка; он состоит из множества обязательных и необязательных частей:
- **Обязательные:** Фундамент, стены, крыша.
- **Необязательные:** Окна, двери, гараж, бассейн, система отопления.
Создание такого объекта требует учета всех этих компонентов, причем в определенном порядке и с определенными зависимостями.
#### 2.Проблемы "монструозного" конструктора (Telescoping Constructor)
Первое, что приходит в голову при попытке создать такой объект без паттернов — это использовать один большой конструктор с множеством параметров:

```
// Антипаттерн: "Телескопический конструктор"
Дом(фундамент, стены, крыша, окна, двери, гараж, бассейн, отопление) { ... }
```

Такой подход приводит к ряду проблем:
- **Сложность вызова:** Вызов такого конструктора выглядит громоздко и непонятно, особенно если некоторые параметры `null` (необязательные).
- **Трудночитаемость:** Порядок следования параметров критичен. Легко перепутать, где должен быть бассейн, а где отопление, что приводит к ошибкам.
- **Взрыв количества конструкторов:** Чтобы поддерживать разные комбинации (дом без гаража, дом без бассейна), нам пришлось бы создавать множество перегруженных конструкторов. Код превращается в лапшу из похожих конструкторов.

#### 3.Проблема неконсистентного состояния при использовании сеттеров
Вторая попытка решить проблему — использовать конструктор по умолчанию (без параметров) и затем вызывать методы-сеттеры (`set...`):
```
Дом дом = new Дом();
дом.setФундамент(новый Фундамент());
дом.setСтены(новые Стены());
// Забыли установить крышу
дом.setОкна(новые Окна());
```
Это решает проблему длинного конструктора, но создает новую, более опасную проблему:
- **Неконсистентное (неполное) состояние:** Объект `дом` может существовать в неполном состоянии. Если часть кода попытается использовать этот `дом` до того, как ему установили крышу или стены, программа, скорее всего, упадет с ошибкой (например, `NullPointerException`).
- **Отсутствие гарантий:** Ничто не мешает разработчику забыть вызвать нужный сеттер.
#### 4.Иллюстрация: "Спагетти-код" создания объекта
Оба подхода ведут к так называемому "спагетти-коду" в клиентской части приложения — там, где мы создаем объект:
- Либо у нас огромная, нечитаемая строка вызова конструктора.
- Либо у нас куча разрозненных вызовов сеттеров, разбросанных по разным строкам, между которыми может выполняться другая логика.
**Вывод:** Существующие стандартные способы (конструкторы и сеттеры) не справляются элегантно с задачей создания **сложных** объектов, особенно когда процесс создания состоит из **шагов**, а результат должен быть **целостным** и **валидным**. Именно здесь на помощь приходит паттерн "Строитель".
### Слайд 3: Решение: Идея паттерна Builder (Строитель)

#### 1.Разделение процесса создания сложного объекта на отдельные шаги
Ключевая идея паттерна "Строитель" заключается в том, чтобы **вынести логику создания объекта из его класса в отдельный класс-строитель**.
Вместо того чтобы пытаться создать объект за один вызов конструктора, мы разбиваем этот процесс на серию последовательных, независимых шагов (методов).
Например, создание дома превращается в такие шаги:
- `построитьФундамент()`
- `возвестиСтены()`
- `установитьКрышу()`
- `вставитьОкна()`

Эти методы вызываются в правильном порядке, и только после завершения всех необходимых шагов клиент получает готовый, **валидный** и **целостный** объект.

#### 2.Аналогия из жизни: прораб (Builder) координирует строительство дома по плану
Отличная аналогия для понимания паттерна — это процесс строительства дома в реальной жизни:
- **Клиент (Заказчик)**: Это вы. Вы хотите получить готовый дом, но вы не хотите лично класть кирпичи или мешать бетон.
- **Продукт (Product)**: Сам готовый дом.
- **Строитель (Builder/Прораб)**: Это главный строитель или прораб. Он знает точную последовательность действий и отвечает за качество каждого этапа. Вы даете прорабу указания, что хотите (например, "построить дом с гаражом"), а он уже координирует рабочих и выполняет все шаги.

Клиент взаимодействует с прорабом (Строителем), а не с отдельными рабочими (которые реализуют детали постройки стен, крыши и т.д.). Прораб гарантирует, что дом будет построен по проекту и будет пригоден для жилья.

#### 3.Базовая схема: `Client` -> `Builder` -> `Product`
Эта схема иллюстрирует взаимодействие между основными участниками паттерна:
- **`Client` (Клиент)**: Это ваш код, который инициирует процесс создания. Клиент создает экземпляр `Builder`, вызывает нужные методы пошаговой сборки (например, `.установитьСтены()`, `.установитьОкна()`) и в конце запрашивает готовый `Product` (через метод `getResult()` или `build()`).
- **`Builder` (Строитель)**: Это класс, который "знает", как собрать `Product`. Он содержит методы для пошагового конструирования и хранит ссылку на создаваемый объект (или его части).
- **`Product` (Продукт)**: Это сложный объект, который в итоге получается. Класс `Product` остается чистым и не содержит логики своей сборки.
Это разделение обязанностей делает код чище, понятнее и безопаснее, гарантируя, что клиент не получит неполный объект.
